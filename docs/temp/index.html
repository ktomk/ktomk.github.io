1. Initializing

Example 1. Mark the Beginning of the Epoch

After the init is the first commit, best at once:

$ git init
Initialized empty Git repository in /home/.../.git/

$ GIT_AUTHOR_DATE="Thu, 01 Jan 1970 00:00:00 +0000" \
  GIT_COMMITTER_DATE="Thu, 01 Jan 1970 00:00:00 +0000" \
    git commit --allow-empty -m 'Initial commit'
[master (root-commit) 1c6d498] Initial commit

No earlier commit will ever be possible (by time and for this root). Simple is better (for comparison an earlier Empty Gitignore Variant).

5.3. GNU find and git-ls-files

  • make appendix

GNUs find and git ls-files can be made inter-operable to large extend for listing of file-paths and usage patterns like find …​ | xargs …​ replacing find …​ with git …​ or by extending with find listings and many more. Some cases for reference, as differences are often in detail. Some caveats.

Leading Directory in Path
$ find -type f,l | head -1
./.gitignore

$ git ls-files | head -1
.gitignore

$ find sandbox -type f | head -1
sandbox/php/flatten.php

$ git ls-files sandbox | head -1
sandbox/.gitattributes

These examples highlight mainly two points important for output: Path output can be different and ordering can be different.

Sort and Path Variants
$ find -type f,l -printf '%P\n' | head -1 (1)
.gitignore

$ find -type d -path './.git' -prune -o -type f -printf '%P\n' | head -1 (2)
.gitignore

$ git ls-files -oc | head -1 (3)
.idea/.gitignore (4)

$ find -type f,l -printf '%P\n' | head -1
php/flatten.php

$ find -type f,l -printf '%P\n' | LC_ALL=C sort | head -1 (5)
.gitattributes

$ git ls-files sandbox | head -1
sandbox/.gitattributes (6)

$ git -C sandbox ls-files | head -1
.gitattributes (7)
1 Note find -printf %P option argument to remove the ./ directory prefix.
2 Note find -prune option to exclude the Git directory.
3 Note git-ls-files (with --other [--cached] [--ignored] options) still is superior listing files when .git/…​ is not the answer.
4 Adding -o (, --other) has different sorting, other files are first
5 sort in the C locale to streamline with sorting cached files in git-ls-files and alike

GNU Find as a Filter for git-ls-files
$ find -type l -o \( -type f -size 0 -o -name '.?*' \) | wc -l
13

$ git ls-files -z | sed -zn '/\(^\|\/\)[A-Z]\+$/Ip' \
    | xargs -0rI '{}' find -L '{}' -type f -size +0 \
    | LC_ALL=C sort
COPYING
sandbox/CÖPYING
sandbox/script

If not otherwise possible, it is an option to pass a file list from git-ls-files — here filtered by sed — into find to process it further. For example to filter out empty files and to resolve symbolic links to regular files (which must not be empty).

This is expensive in execution as xargs will fork find once per file argument. Alternatives may involve git ls-tree -rl --full-tree HEAD if operating on other files is not necessary.

Detecting broken symbolic links is something only the shell can do, not git-ls-files. Which makes sense as symbolic links always need a checkout and may not even resolve within a repository. Git only needs to know how to create the file on Git checkout.

5.5. Case Sensitivity and Filenames

The fun starts with case-in-sensitive file systems (like FAT for example). The file can be renamed to a different case while it stays the same in the file system. Git however is case-sensitive, and it is a new file unless git-config core.ignorecase is true, then Git tries to keep the file under the old name.

The hell breaks loose when core.ignorecase is set to false with a case-in-sensitive file system. A prank of a git.

Appendix B: Non-Portable File- and Pathnames

Even for simple commands they can make a difference: non-portable file-names.

On an Apple macOS system, Gits quoted file- and pathnames may occasionally interfere processing with standard utilities not having options to work with NUL terminators (e.g. not having the GNU Coreutils available).

  • give GNU example for the utility there is (another Appendix for utility options, now mind Macos)

So even git itself has these options to use a null-byte instead of a new-line as terminator/separator, sed or sort may not have options like -z, so a common solution is out of reach.

This in itself is already an example why portable file- and pathnames are crucial for interoperability especially when sharing files across many systems like with a Git repository. They are just asking for a multitude of problems.

Better prevent non-portable names for files in a repository and fix those names by renaming if they have been introduced.

B.1. Git File-Name Quoting

As far as git itself is concerned, the file-name quoting of Git with git-ls-files, git-ls-tree and git-grep etc. can be used to detect files with non-portable characters.

Git automatically quotes those files that contain "odd" characters in their name, for example names containing a newline or tab, or those having characters in the upper-half of a byte.

Those file-names will be wrapped into double-quotes having C-escapes for the non-portable characters:

Example 11. Git File-Name Quoting

CÖPYING (quoted) → "C\303\226PYING"

The UTF-8 binary sequence 0xC3 0x96 of the Unicode Character 'LATIN CAPITAL LETTER O WITH DIAERESIS' (U+00D6) in the octal number representation like in an Escape Sequence in C when quoted.

fu
es      ops he (quoted) → "fu\n s\tops he\re"

Control characters like new-line, tab or carriage-return in their representation of an Escape Sequence in C when quoted.

The quoting with double-quotes then sometimes prevents straight-forward processing or stands in the way. Using a NUL terminator option (-z) makes git providing the original binary encoding without the quotes and terminating by a null-byte (NUL) instead of a line-feed (LF).

This principle is shared across utilities on Linux as most of them can process NUL-terminated instead of LF-terminated inputs and/or can create such a kind of output.

This may not hold for platforms other than Linux. Therefore, it is good to keep an eye on portable filenames and not introduce non-portable filenames into a system, specifically into a Git repository as the platform it will be checked out - now or in future - might be different to the platform or file-system where a file was checked in.

As git does the legwork quoting such filenames, they are easy to list:

Example 12. List Quoted File- and Pathnames
$ git ls-files | sed -n '/^"/p'
"C\303\226PYING"
"fu\n s\tops he\re"

While control characters would hinder the output, for everything else the following alternative may be of interest:

$ git ls-files | sed -n '/^"/p' | xargs printf "%b\n"
CÖPYING

Producing any output is a sign that the repository already contains some non-portable filenames, but this is not the whole story.

B.2. Problems Detecting Non-Portable Names

Git does not quote all non-portable file-names, for example those staring with a hyphen-minus or containing spaces. So looking for quotes alone is only part of the story.

Likely this is not Gits fault as the quoting done is enough for Gits output (and quoting all non-portable pathnames would not help in terms of shell-reuse).

This is a reminder that there is more in portable-filenames than just control or "odd" characters but also common ones just by their position and quoting. Like the minus in front of a file-name.

Example 13. Incomplete Listing of Non-Portable Pathnames
$ git ls-files | sed -n '/^[-"]/p'
--help
"C\303\226PYING"
"fu\n s\tops he\re"

Despite this listing includes filenames starting with a hyphen-minus, it is still incomplete because pathnames including spaces are not specifically reported here.

So there are multiple rules to be checked for filename portability (e.g. not starting with a hyphen-minus; only characters from A-Z, a-z, 0-9, underscore, hyphen-minus and dot) and doing this with sed or grep runs already out of practical options.

D.2. Bash Glob and Git Pathspec

Gits pathspec [7] * matches the directory separator as well. This is normally of benefit, especially as everything up to the last slash represents a directory prefix, it is most often quick and easy (short) to write a pattern.

To make a pathspec pattern more glob-like, the special meaning word glob makes it so.

Appendix E: Shell Notes

E.1. LC_ALL=C

The LC_ALL environment parameter controls the locale of diverse lib-c based utilities, the way they sort or treat strings the same or not. Also, how patterns match, for example character-classes like [a-z] in a regular expression.

Setting the locale to C by prefixing a simple or complex shell command with LC_ALL=C makes the command portable accross different shells as it overrides the standard settings in the users shell.

The C locale aligns well with the default sort-order of git-ls-files.
